Разработка:
  Клонирую репозиторий библиотеки и создаю новую ветку. Вношу нужные изменения. Потом делаю commit и push.

Тестирование:
     Подключаю библиотеку в проект через composer (изменяю старую версию на новую в composer.json).
     Обновляю зависимости: composer update названия библиотеки.
     Провожу тестирование в проекте (вручную или автотесты).

Релиз библиотеки:
     Мержу изменения в главную ветку. Потом ставлю тег новой версии библиотеки — например:
    git tag v1.0.1 && git push origin v1.0.1.

Обновление проекта:
     Уже по новым composer.json и composer.lock файлам, которые я получил в результате тестирования обновлённой библиотеки,
    делаю commit и push — тоже в новой ветке.

Релиз и деплой проекта:
     Делаю merge новой ветки в главную. Потом я бы тегировал уже новую версию проекта и пушил, например:
    git tag v1.22.3 && git push origin v1.22.3, так как это, по-моему, хорошая практика.
    Так мы зафиксируем момент релиза и сможем легко вернуться к предыдущему, если что-то пойдёт не так после релиза,
    либо можно, как второй вариант, использовать релизные ветки с номерами.

    Потом деплой через CI/CD, либо, если он не настроен — вручную через git pull.
